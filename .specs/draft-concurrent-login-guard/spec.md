# 동시 로그인 세션 제어 (Concurrent Login Session Guard)

> **Source Issue**: [FR-2189](https://lablup.atlassian.net/browse/FR-2189)

## 개요

사용자가 올바른 자격증명으로 로그인을 시도할 때, 다른 위치에서 이미 활성화된 로그인 세션이 존재하는 경우 이를 감지하여 사용자에게 알리고, 기존 세션을 종료한 후 새 세션으로 진행할지 확인하는 기능이다. 비밀번호 만료(password max-age) 플로우와의 우선순위 충돌도 고려하여, 두 조건이 동시에 발생하는 경우의 처리 순서를 정의한다.

## 문제 정의

현재 WebUI는 동일 사용자가 여러 브라우저/디바이스에서 동시에 로그인 세션을 유지할 수 있다. 이로 인해:

- 보안 관점에서 세션 탈취나 자격증명 공유를 감지하기 어렵다.
- 관리자가 사용자별 동시 접속을 제어할 수 없다.
- 사용자가 의도하지 않은 다른 위치의 세션이 유지되고 있는지 알 수 없다.

## 가정 (Assumptions)

> 아래 가정은 [Open Questions](#open-questions) 섹션의 답변에 따라 변경될 수 있다.

1. **서버 API 변경 선행**: 백엔드(Manager)의 로그인 API가 "이미 활성 세션 존재" 상태를 `fail_reason`으로 반환하도록 변경된다고 가정한다. WebUI는 이 `fail_reason`을 기반으로 분기 처리한다.
2. **SESSION 모드 전용**: 이 기능은 SESSION 연결 모드에서만 적용된다. API 모드는 키페어 기반 인증이므로 동시 세션 개념이 적용되지 않는다.
3. **기존 세션 종료 후 처리**: 기존 세션이 서버에서 무효화되면, 해당 위치에서는 다음 네트워크 요청 시 기존 401 핸들링 로직(RelayEnvironment의 AuthorizationError + LoginSessionExtendButton 만료 감지)이 자동으로 동작하여 로그인 화면으로 전환된다.

## 요구 사항

### Must Have

- [ ] 사용자가 올바른 자격증명으로 로그인 시도 시, 서버가 "이미 활성 세션 존재"를 응답하면 확인 다이얼로그를 표시한다.
- [ ] 확인 다이얼로그에 "다른 위치에서 이미 활성 세션이 있습니다. 기존 세션을 종료하고 여기에서 로그인하시겠습니까?" 메시지를 표시한다.
- [ ] 사용자가 "확인(진행)"을 선택하면 기존 세션을 종료하고 새 로그인을 진행한다.
- [ ] 사용자가 "취소"를 선택하면 로그인을 중단하고 로그인 폼으로 돌아간다. 입력한 자격증명은 폼에 유지된다.
- [ ] 비밀번호 만료 + 동시 세션이 동시에 발생하는 경우: 비밀번호 변경을 먼저 진행한 후, 동시 세션 확인 다이얼로그를 표시한다.
- [ ] 기존 세션이 강제 종료된 위치에서는 다음 API 요청 시 세션 만료 처리가 자동으로 동작한다 (기존 401 핸들링 로직 활용).

### Nice to Have

- [ ] 기존 세션이 강제 종료된 위치에서 "다른 위치에서 로그인하여 세션이 종료되었습니다" 등 구체적인 종료 사유 메시지를 표시한다 (기존 "session expired"와 차별화).
- [ ] 확인 다이얼로그에 기존 세션의 접속 정보(마지막 접속 시간, IP 등)를 표시한다.
- [ ] 관리자 설정으로 동시 세션 제어 기능을 활성화/비활성화할 수 있다.

## 사용자 시나리오

### 시나리오 1: 기본 동시 세션 감지 및 처리

**전제**: 사용자 A가 디바이스 1에서 이미 로그인된 상태

1. 사용자 A가 디바이스 2에서 로그인 폼에 올바른 이메일/비밀번호를 입력하고 "로그인" 버튼을 클릭한다.
2. 서버가 "이미 활성 세션 존재" 응답을 반환한다.
3. WebUI가 확인 다이얼로그를 표시한다:
   - 메시지: "다른 위치에서 이미 활성 세션이 있습니다. 기존 세션을 종료하고 여기에서 로그인하시겠습니까?"
   - 버튼: "로그인 진행" / "취소"
4. 사용자가 "로그인 진행"을 선택한다.
5. 서버에서 디바이스 1의 세션을 무효화하고, 디바이스 2에서 새 세션을 생성한다.
6. 디바이스 2에서 정상적으로 로그인이 완료된다.
7. 디바이스 1에서 다음 API 요청(Relay GQL 쿼리, 세션 연장 등) 시 401 응답을 받고, 기존 세션 만료 처리 로직에 의해 로그인 화면으로 전환된다.

### 시나리오 2: 동시 세션 로그인 취소

**전제**: 사용자 A가 디바이스 1에서 이미 로그인된 상태

1. 사용자 A가 디바이스 2에서 로그인을 시도한다.
2. 확인 다이얼로그가 표시된다.
3. 사용자가 "취소"를 선택한다.
4. 로그인 폼으로 돌아간다. 이메일과 비밀번호 필드에는 기존 입력값이 유지된다.
5. 디바이스 1의 세션은 영향을 받지 않는다.

### 시나리오 3: 비밀번호 만료 + 동시 세션 동시 발생

**전제**: 사용자 A의 비밀번호가 만료되었고, 디바이스 1에서 이미 로그인된 상태

1. 사용자 A가 디바이스 2에서 로그인을 시도한다.
2. 서버가 "비밀번호 만료" 응답을 반환한다.
3. 비밀번호 변경 모달(`ResetPasswordRequiredInline`)이 표시된다.
4. 사용자가 새 비밀번호를 입력하고 변경을 완료한다.
5. 변경된 비밀번호로 자동 재로그인을 시도한다.
6. 서버가 "이미 활성 세션 존재" 응답을 반환한다.
7. 동시 세션 확인 다이얼로그가 표시된다.
8. 사용자가 "로그인 진행"을 선택하면 기존 세션 종료 후 새 세션으로 로그인이 완료된다.

### 시나리오 4: TOTP + 동시 세션 동시 발생

**전제**: 사용자 A에게 TOTP가 활성화되어 있고, 디바이스 1에서 이미 로그인된 상태

1. 사용자 A가 디바이스 2에서 이메일/비밀번호를 입력하고 로그인을 시도한다.
2. 서버가 TOTP 코드를 요청한다.
3. 사용자가 OTP 코드를 입력하고 다시 로그인을 시도한다.
4. 서버가 "이미 활성 세션 존재" 응답을 반환한다.
5. 동시 세션 확인 다이얼로그가 표시된다.
6. 이후 플로우는 시나리오 1과 동일하다.

### 시나리오 5: SSO(SAML/OpenID) 로그인과 동시 세션

**전제**: 사용자 A가 디바이스 1에서 이미 로그인된 상태

1. 사용자 A가 디바이스 2에서 SAML/OpenID 로그인을 시도한다.
2. SSO 인증 완료 후 콜백으로 돌아올 때, 서버가 동시 세션 상태를 반환한다.
3. 동시 세션 확인 다이얼로그가 표시된다.
4. 이후 플로우는 시나리오 1과 동일하다.

> **참고**: SSO 플로우는 서버 측 리다이렉트 기반이므로, 동시 세션 감지가 콜백 시점에서 가능한지 백엔드 확인이 필요하다. ([Open Question 4](#open-questions) 참조)

## 수락 기준 (Acceptance Criteria)

- [ ] SESSION 모드에서 이미 활성 세션이 있는 상태로 동일 사용자가 로그인 시도 시, 확인 다이얼로그가 표시된다.
- [ ] 확인 다이얼로그에서 "로그인 진행" 선택 시, 기존 세션이 종료되고 새 로그인이 성공적으로 완료된다.
- [ ] 확인 다이얼로그에서 "취소" 선택 시, 로그인 폼으로 돌아가며 기존 입력값이 유지된다.
- [ ] 기존 세션이 강제 종료된 위치에서 다음 네트워크 요청 시 로그인 화면으로 전환된다.
- [ ] 비밀번호 만료 + 동시 세션이 동시에 발생할 때, 비밀번호 변경이 먼저 처리된 후 동시 세션 확인이 진행된다.
- [ ] 확인 다이얼로그는 i18n을 지원한다.
- [ ] 로딩 상태: 사용자가 "로그인 진행"을 선택한 후 기존 세션 종료 + 새 로그인 완료까지 로딩 인디케이터가 표시된다.

## 범위 외 (Out of Scope)

- 관리자가 사용자별로 허용되는 최대 동시 세션 수를 설정하는 기능 (이 스펙에서는 동시 세션 수 1로 고정)
- 활성 세션 목록 조회 및 개별 세션 관리 UI
- API 모드에서의 동시 세션 제어
- 웹소켓/실시간 푸시를 통한 기존 세션 즉시 종료 알림 (기존 세션은 다음 API 요청 시점에 만료 처리)
- Backend Manager 서버 측 API 변경 (별도 이슈로 관리)

## Open Questions

1. **동시 세션 감지 API**: 서버(Manager)가 로그인 API 응답의 `fail_reason`으로 "이미 활성 세션이 있음"을 반환할 예정인가? 반환 형식은? (예: `fail_reason: "Another active session exists"`, `data: { session_id, last_access, ip }`) WebUI에서 별도 API를 호출해야 하는가?

2. **적용 범위 확인**: SESSION 모드에서만 적용하는 것이 맞는가? API 모드도 고려해야 하는가?

3. **기존 세션 종료 위치의 메시지**: 기존 세션이 강제 종료된 위치에서 단순 "session expired" 처리로 충분한가? "다른 위치에서 로그인하여 세션이 종료되었습니다" 등의 구체적 메시지가 필요한가? (구체적 메시지를 위해서는 서버가 401 응답에 종료 사유를 포함해야 할 수 있음)

4. **SSO 플로우 호환성**: SAML/OpenID 로그인은 서버 리다이렉트 기반으로 동작하는데, 콜백 시점에서 동시 세션 감지 + 사용자 확인 플로우를 구현할 수 있는가? (서버 측 확인 필요)

5. **기존 세션 강제 종료 메커니즘**: "로그인 진행" 선택 시 서버에 보내는 요청은 기존 `client.login()` 호출에 force 파라미터를 추가하는 방식인가? 별도 세션 종료 API를 먼저 호출한 후 재로그인하는 방식인가?
