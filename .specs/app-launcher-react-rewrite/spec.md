# App Launcher React Rewrite

## Overview

Rewrite the existing `EduAppLauncher` component and `/applauncher` route to follow modern React patterns. The current implementation uses legacy `globalThis.backendaiclient` patterns with imperative code. The goal is to make it idiomatic React with proper hooks, state management, and UI feedback.

## Background

The `/applauncher` route enables external portals (e.g., education portals) to launch specific apps (JupyterLab, etc.) for Backend.AI sessions via URL parameters. An admin can generate a URL with an sToken (HMAC-signed credential), and when a user clicks the link, the WebUI:

1. Authenticates via the sToken (token-based login)
2. Finds or creates a compute session
3. Launches the requested app (e.g., JupyterLab) and redirects the user to it

### Example URL

```
/applauncher?sToken=BackendAI+signMethod%3DHMAC-SHA256%2C+credential%3DAKIAIOSFODNN7EXAMPLE%3A...&api_version=v7.20230615&date=2023-08-15+13%3A49%3A13.888434%2B00%3A00&endpoint=10.20.30.10%3A8081&session_id=d55813d5-ea62-4678-9d02-d9f078c2d28d&app=jupyterlab
```

### URL Parameters

| Parameter | Required | Description |
|-----------|----------|-------------|
| `sToken` | Yes | HMAC-SHA256 signed credential for authentication |
| `api_version` | No | Backend.AI API version |
| `date` | No | Request timestamp |
| `endpoint` | No | Backend.AI manager endpoint |
| `session_id` | No | Existing session ID to launch app on (if omitted, creates new session) |
| `app` | No | App to launch (default: `jupyter`). e.g., `jupyterlab`, `jupyter`, `vscode` |
| `session_template` / `sessionTemplate` | No | Session template name for new session creation |
| `cpu` / `mem` / `shmem` / `cuda-shares` / `cuda-device` | No | Resource allocation for new sessions |

## Current State (Problems)

1. **Legacy client pattern**: Uses `globalThis.BackendAIClient` / `globalThis.backendaiclient` directly instead of React context/hooks
2. **No UI feedback**: The component renders `<></>` (empty fragment) with no loading/error/progress states visible to the user
3. **Imperative flow**: All logic is in a single `_launch()` function chain inside `useEffect`, not decomposed into React-friendly hooks
4. **Duplicated proxy code**: `appLauncherProxy.ts` duplicates logic from `useBackendAIAppLauncher.tsx`
5. **No error boundary integration**: Uses custom `_dispatchNotification` instead of React error patterns
6. **Missing React Router integration**: Doesn't use `useSearchParams` for URL parameter handling

## Target State

### Architecture

```
/applauncher route
  -> AppLauncherPage (page component)
    -> useAppLauncherParams() - parse & validate URL search params
    -> AppLauncherFlow (flow orchestrator component)
      -> Step 1: Token Authentication (useTokenAuth hook)
      -> Step 2: Session Resolution (find existing or create new)
      -> Step 3: App Launch (reuse useBackendAIAppLauncher or appLauncherProxy)
      -> UI: Loading states, progress, error display
```

### Key Design Decisions

1. **Use `useSearchParams`** from React Router to parse URL parameters
2. **Step-based UI**: Show the user which step they're on (Authenticating -> Finding Session -> Launching App)
3. **Proper error handling**: Display errors inline with retry options, not just notifications
4. **Reuse existing proxy infrastructure**: Use `appLauncherProxy.ts` (or `useBackendAIAppLauncher`) for the actual app launch, don't duplicate
5. **Keep the legacy client init** for token login since the sToken auth flow requires the imperative client (this is an external auth flow, not the normal login flow)

### UI Design

A centered card/panel showing:
- Current step with progress indicator
- Step descriptions: "Authenticating...", "Finding session...", "Launching JupyterLab..."
- Error state with message and retry button
- Auto-redirect when app URL is ready

### Scope

- Rewrite `EduAppLauncher.tsx` and `EduAppLauncherPage.tsx`
- Keep both `/applauncher` and `/edu-applauncher` routes working
- Add proper TypeScript types for URL parameters
- Add loading/error/success UI states
- Reuse existing `appLauncherProxy.ts` for proxy operations
- Keep backward compatibility with all existing URL parameter formats

### Out of Scope

- Changing the sToken authentication mechanism itself
- Modifying `appLauncherProxy.ts` or `useBackendAIAppLauncher.tsx` internals
- Adding new URL parameters or features

## Backend.AI API Reference

### Token-based Authentication

**Endpoint**: `POST /server/token-login`

The sToken is an HMAC-SHA256 signed credential generated by external portals. The token format is:
```
BackendAI signMethod=HMAC-SHA256, credential=<access_key>:<signature>
```

The signature is computed using a double-HMAC approach:
1. Derive a signing key from the secret key + date: `HMAC-SHA256(secret_key, date_str)`
2. Sign the request string: `HMAC-SHA256(signing_key, hostname)`

The token can be passed as:
- URL parameter (`sToken` or `stoken`, case-insensitive)
- Cookie (`sToken`)
- The token name is configurable via `config.api.auth_token_name`

**WebUI client flow**: The `BackendAIClient` handles token login via `client.token_login(sToken, extraParams)`, which sends the token to the manager for session-based authentication.

### Session Creation from Template

**Endpoint**: `POST /session/_/create-from-template`

Request body:
```json
{
  "template_id": "<session-template-uuid>",
  "session_name": "<optional-name>",
  "image": "<optional-image-override>",
  "group_name": "<project-name>",
  "domain_name": "<domain-name>",
  "scaling_group": "<resource-group>",
  "bootstrap_script": "<optional-startup-script>",
  "mounts": ["<vfolder-name>", ...],
  "resources": { "cpu": "1", "mem": "1g", ... }
}
```

The WebUI client wraps this via `client.createSessionFromTemplate(templateId, null, null, sessionResources, timeout)`.

### Session Template Listing

**Endpoint**: `GET /session_template/`

Returns all session templates. The WebUI filters by name to find the matching template for the requested app.

### App Launch (Start Service)

**Endpoint**: `POST /session/{session_name}/start-service`

Request body:
```json
{
  "service": "<app-name>",
  "port": <optional-port>,
  "open_to_public": <boolean>,
  "envs": { ... },
  "arguments": { ... }
}
```

This is handled by the wsproxy infrastructure in `appLauncherProxy.ts`:
- **V1 proxy**: Configures proxy via `PUT /conf`, then opens `proxy/{token}/{session_id}/add?app=<name>`
- **V2 proxy**: Calls `computeSession.startService()` which hits the manager, then opens `v2/proxy/{token}/{session_id}/add?app=<name>`

### Session Listing

**Endpoint**: GraphQL `compute_session_list` query

Used to find existing sessions matching the template's image. Fields queried: `session_id`, `name`, `access_key`, `status`, `status_info`, `service_ports`, `mounts`.

### Education API Helpers

The `backendaiclient.eduApp` namespace provides:
- `get_mount_folders()` — Get folders to auto-mount for edu sessions
- `get_user_projects()` — Get available projects for the user
- `get_user_credential(sToken)` — Get bootstrap script for user credential setup

## E2E Testing Strategy

### Goal

Test the full `/applauncher` flow end-to-end: from URL navigation with sToken → authentication → session creation/resolution → app launch and redirect. This validates the complete Edu App Launcher experience that external education portals rely on.

### Infrastructure Requirements

Running the full E2E flow requires:

1. **Backend.AI Cluster** (Manager + Agent + AppProxy)
   - Manager with session template API support
   - At least one agent with available compute resources
   - AppProxy (coordinator + worker) for app proxying
2. **Valid sToken Credentials**
   - An access key + secret key pair for HMAC-SHA256 signing
   - The E2E test must generate sTokens programmatically (see sToken Generation below)
3. **Session Template**
   - At least one session template registered in the Backend.AI manager
   - The template image must be available on the agent
4. **WebUI Dev Server** running and configured to point at the Backend.AI cluster
5. **wsproxy** running for proxy connectivity

### sToken Generation for E2E Tests

The sToken is an HMAC-SHA256 signed credential. To generate it in E2E tests:

```typescript
import crypto from 'crypto';

function generateSToken(accessKey: string, secretKey: string, endpoint: string): {
  sToken: string;
  apiVersion: string;
  date: string;
} {
  const now = new Date();
  const dateStr = now.toISOString().split('T')[0]; // YYYY-MM-DD
  const fullDate = now.toISOString().replace('T', ' ').replace('Z', '+00:00');

  // Extract hostname from endpoint
  const url = new URL(endpoint.startsWith('http') ? endpoint : `http://${endpoint}`);
  const hostname = url.hostname;

  // Double-HMAC: signingKey = HMAC(secretKey, dateStr), signature = HMAC(signingKey, hostname)
  const signingKey = crypto.createHmac('sha256', secretKey).update(dateStr).digest();
  const signature = crypto.createHmac('sha256', signingKey).update(hostname).digest('hex');

  const sToken = `BackendAI signMethod=HMAC-SHA256, credential=${accessKey}:${signature}`;

  return {
    sToken,
    apiVersion: 'v7.20230615',
    date: fullDate,
  };
}
```

> **Note**: The exact signing algorithm must match the Backend.AI manager's token verification. See `src/ai/backend/manager/` in the backend.ai repo for the canonical implementation.

### Test Tiers

#### Tier 1: UI-Only Tests (No Backend Required)
Already implemented in `e2e/app-launcher/app-launcher-page.spec.ts`:
- Missing sToken error state
- Step-based UI rendering (3 steps visible)
- Loading spinner display
- Retry button functionality
- Route alias (`/edu-applauncher`) support

#### Tier 2: Authentication Tests (Backend Required, No Session)
Tests that require a running Backend.AI manager but don't create sessions:
- Valid sToken → successful authentication (step 1 passes, step 2 may fail without template)
- Invalid sToken → authentication error with proper message
- Expired sToken → authentication error
- Missing endpoint fallback to localStorage/config

#### Tier 3: Full Flow Tests (Full Cluster Required)
Tests that exercise the complete flow including session creation and app launch:

**3.1 Direct Session ID Flow**
```
/applauncher?sToken=<valid>&session_id=<running-session>&app=jupyterlab
```
- Authenticate → skip template resolution → launch app → redirect to proxy URL

**3.2 Template-based Session Creation Flow**
```
/applauncher?sToken=<valid>&session_template=<template-name>&app=jupyterlab
```
- Authenticate → list sessions → no match → create from template → wait for RUNNING → launch app → redirect

**3.3 Template-based Session Reuse Flow**
```
/applauncher?sToken=<valid>&session_template=<template-name>&app=jupyterlab
```
- Authenticate → list sessions → find matching RUNNING session → launch app → redirect

**3.4 Error Scenarios**
- Non-existent session template → `NoSessionTemplate` error
- Image mismatch between session and template → `CannotCreateSessionWithDifferentImage` error
- Session in non-RUNNING status → `SessionStatusIsWithReload` error
- Timeout during session creation → `SessionStillPreparing` error
