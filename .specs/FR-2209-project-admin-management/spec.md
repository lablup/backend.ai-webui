# WebUI 프로젝트 관리자 기능

> **원본 이슈**: GF-30, GF-31, GF-32 (에픽: GF-107)
> **상태**: 초안 (4차 리뷰 반영)
> **작성일**: 2026-03-04
> **최종 수정**: 2026-03-26

## 요약

RBAC 기반의 프로젝트 관리자(Project Admin) 역할을 WebUI에 반영합니다. 프로젝트 관리자는 자신이 할당된 프로젝트 내에서 연산 세션, 모델 서비스, 프로젝트 폴더를 관리하고 프로젝트 멤버를 조회할 수 있습니다. **초기 버전에서는 사용자 mutation(생성/추가/삭제)은 제외되며 조회 위주로 제공됩니다.** 다른 사용자의 세션·서비스에 대한 액션 범위는 현재 superadmin과 동일하며, 프로젝트 폴더는 프로젝트 범위에 한해 CRUD가 가능합니다. 전역 사용자 목록 조회나 다른 사용자의 개인 폴더 접근은 불가합니다.

**UI 레이아웃 전략 (확정)**: 일반 사용자 페이지는 사용자 본인의 리소스만 표시합니다 (프로젝트 관리자 역할이 있더라도). 프로젝트 관리자 기능은 별도의 **Admin 메뉴**에 전용 세션/서비스/폴더 목록으로 제공됩니다. 단, Admin 메뉴 분리 자체는 **별도 에픽/스펙으로 진행**하며, 이 스펙에서는 각 기능의 요구사항만 정의합니다.

## 배경

프로젝트가 많은 대규모 조직에서는 superadmin 혼자서 모든 프로젝트를 관리하기 어렵습니다. Backend.AI 26.3.0에 도입된 RBAC 시스템은 **프로젝트 범위의 관리자 역할**을 지원하여, 지정된 사용자가 전역 superadmin 권한 없이도 자신이 할당된 프로젝트 내에서 리소스를 관리할 수 있게 합니다.

### RBAC 모델 (백엔드)

백엔드 RBAC 시스템은 프로젝트마다 두 가지 시스템 역할을 생성합니다:
- **프로젝트 관리자** (`role_project_<id>_admin`): 프로젝트 범위 내 모든 리소스 엔티티 타입에 대한 전체 권한 (create, read, update, soft-delete, hard-delete, grant:*)
- **프로젝트 멤버** (`role_project_<id>_member`): 프로젝트 범위 내 읽기 전용 권한

프로젝트 관리자와 관련된 엔티티 타입:
- `SESSION` — 연산 세션
- `MODEL_DEPLOYMENT` / `DEPLOYMENT` — 모델 서비스
- `VFOLDER` — 가상 폴더
- `USER` — 프로젝트 내 사용자 가시성

### 용어 매핑

| 이 문서의 용어 | WebUI 코드 (`useCurrentUserRole()`) | 설명 |
|---------------|-------------------------------------|------|
| 프로젝트 관리자 (Project Admin) | (RBAC 기반, 신규) | 프로젝트 범위 관리자. `my_roles` 쿼리로 감지 |
| 도메인 관리자 (Domain Admin) | `admin` | 도메인 범위 관리자 |
| superadmin | `superadmin` | 전역 시스템 관리자 |

### 현재 상태 (WebUI)

- 사용자 역할은 `useCurrentUserRole()`을 통해 `superadmin | admin | user | monitor`로 판별
- WebUI에 프로젝트 범위 관리자 개념이 없음
- Superadmin은 모든 세션/서비스/폴더를 볼 수 있고, 일반 사용자는 자신의 것만 볼 수 있음
- 권한 확인이 역할 문자열 기반이며, RBAC 기반이 아님
- 세션 뷰가 사용자 뷰와 관리자 뷰로 분리된 상태

**범위 외**:
- RBAC 역할/권한 관리 UI (별도 진행 중)
- 백엔드 RBAC 구현
- Admin 메뉴 분리 (별도 에픽/스펙으로 진행)
- 프로젝트 공유 세션 기능 (프로젝트 멤버 누구나 접근 가능한 세션) — 별도 이슈로 진행. 세션 공유/페어 프로그래밍 요구는 있으나 현재 구현 불가 (회의 2026-03-26)
- 공유 서비스(shared services)의 서비스 설정 편집 — 프로젝트 공유 세션과 함께 향후 고려
- 프로젝트 관리자에 의한 사용자 계정 생성 — 리소스 정책·키페어 등 복잡한 의존관계로 인해 초기 버전 범위 외. 향후 RBAC 기반 신규 mutation 준비 시 재검토 (회의 2026-03-26)
- 프로젝트 관리자에 의한 기존 사용자 프로젝트 추가/제거 — 크로스-프로젝트 접근 권한, 프라이버시 이슈 등으로 초기 버전 범위 외 (회의 2026-03-26)
- 프로젝트 관리자에 의한 멤버 역할 변경 — 초기 버전 범위 외, 멤버 목록 조회만 제공 (회의 2026-03-26)
- RBAC 기반 쿼리 API (프로젝트 쿼터 내 사용자별 리소스 사용량 조회) — 백엔드 API 추가 필요, 후속 이슈로 진행
- 프로젝트 관리자 리소스 사용량 뷰 (프로젝트 전체/사용자별 리소스 사용 현황 대시보드) — 후속 이슈로 진행
- 레거시 API 확장을 통한 RBAC 기능 구현 — 레거시 API는 확장하지 않고, RBAC 기반 신규 mutation 사용 (회의 2026-03-26)

## 요구사항

### 선행 조건: 프로젝트 관리자 역할 감지

네 가지 관리 기능을 구현하기 전에, WebUI에서 현재 사용자가 프로젝트 관리자 역할을 갖고 있는지 판별하는 메커니즘이 필요합니다.

**PR-1: 현재 사용자의 프로젝트 관리자 역할 감지**

- **`my_roles` GraphQL 쿼리**를 사용하여 현재 사용자의 RBAC 역할 할당 조회 (백엔드에 추가 완료 확인됨)
- `my_roles` 응답에서 사용자가 특정 프로젝트에 대해 프로젝트 관리자 역할을 갖고 있는지 판별
- React 훅(예: `useCurrentUserProjectRoles()`)을 통해 다음 정보 제공:
  - 사용자가 프로젝트 관리자인지 여부
  - 어떤 프로젝트에 대해 프로젝트 관리자인지
  - 사용 가능한 구체적 권한
- **프로젝트 선택기에 관리자 여부 표시**: 페이지 헤더의 프로젝트 선택기(Select)에서 각 프로젝트 옵션에 해당 사용자가 관리자인지 여부를 시각적으로 표시 (예: 뱃지, 아이콘, 또는 라벨)

**인수 기준:**
- 사용자의 프로젝트 관리자 상태와 프로젝트 범위를 반환하는 훅이 존재할 것
- `my_roles` 쿼리를 호출하여 역할 정보를 조회할 것
- 프로젝트 선택기에서 관리자 역할이 있는 프로젝트가 시각적으로 구분될 것
- `my_roles` 쿼리가 사용 불가능한 경우(RBAC 미지원 코어 버전) 기존 동작으로 우아하게 폴백할 것
- 기존 `useCurrentUserRole()` 동작이 깨지지 않을 것

### 스토리 1: 프로젝트 관리자 — 연산 세션 관리 (GF-30)

프로젝트 관리자는 할당된 프로젝트 내에서 생성된 **모든 연산 세션**을 조회하고 관리할 수 있습니다. 다른 사용자의 세션에 대한 허용/불허 액션은 **현재 superadmin이 다른 사용자의 세션에 대해 갖는 것과 동일**합니다 — 새로운 권한이 추가되거나 제거되지 않으며, 동일한 `isOwner` 체크가 적용됩니다.

> **UI 배치**: 프로젝트 관리자의 세션 관리 기능은 Admin 메뉴의 전용 관리자 세션 뷰에서 제공됩니다. 일반 사용자 페이지의 세션 뷰는 프로젝트 관리자 여부와 관계없이 사용자 본인의 세션만 표시합니다. (현재 세션 뷰는 이미 사용자 뷰와 관리자 뷰로 분리되어 있음)

**기능 요구사항:**

1. **세션 목록 가시성 (Admin 세션 뷰)**
   - 사용자가 프로젝트 P에 대해 프로젝트 관리자 역할을 가진 경우, Admin 메뉴의 관리자 세션 뷰에서 프로젝트 P의 모든 세션을 표시 (자신의 세션뿐만 아니라)
   - 세션 소유자를 구분할 수 있도록 "생성자" / 소유자 컬럼 표시
   - 모든 표시된 세션에 대해 필터링 및 정렬 동작

   > **백엔드 확인 완료**: 프로젝트 관리자가 프로젝트 스코프로 세션 조회 쿼리를 보내면, RBAC 기반으로 해당 프로젝트의 모든 세션이 반환됨. API 호출 전 권한 확인만을 위한 별도 요청(추가 round-trip)은 불필요 — 백엔드가 요청자의 RBAC 권한을 검증하여 결과 반환 또는 에러 응답. 단, UI 레벨의 역할 감지 및 조건부 렌더링은 별도로 유지. (회의 2026-03-26 확인)

2. **세션 제어 작업** (참고: `SessionActionButtons.tsx`, `EditableSessionName.tsx`)

   **다른 사용자의 세션에 허용되는 작업** (`isOwner` 체크 없음):
   - 프로젝트 내 모든 세션 **종료** 가능
   - 프로젝트 내 모든 세션의 **컨테이너 로그 조회** 가능
   - 프로젝트 내 모든 세션의 **리소스 사용량 조회** (CPU, 메모리, GPU) 가능
   - **상태 카테고리별 세션 필터링** (실행 중 / 완료) 가능
   - **세션 선택 및 일괄 작업** (예: 일괄 종료) 가능

   **다른 사용자의 세션에 허용되지 않는 작업** (`isOwner` 필요 — 현재 superadmin도 동일):
   - **앱 런처** — 다른 사용자 세션의 앱 런처 대화상자를 열 수 없음
   - **기본 앱 실행** (Jupyter Notebook / File Browser) — 직접 앱 실행 불가
   - **터미널** — 다른 사용자 세션의 터미널 열기 불가
   - **SFTP 연결** — 다른 사용자 세션의 SFTP 접근 불가
   - **컨테이너 커밋** — 다른 사용자의 세션 상태를 커스텀 이미지로 저장 불가
   - **세션 이름 변경** — 다른 사용자의 세션 이름 변경 불가

3. **UI 동작**
   - 사용자가 여러 프로젝트의 프로젝트 관리자인 경우, 기존 프로젝트 선택기를 사용하여 프로젝트 간 전환하여 세션 조회
   - 세션 소유자가 다른 사용자여도 액션 버튼(종료, 접속 등) 활성화
   - Admin 세션 뷰에 사용자의 상승된 접근 권한 표시 (예: 미묘한 인디케이터 또는 정보 메시지)

**인수 기준:**
- [ ] 프로젝트 관리자가 Admin 세션 뷰에서 프로젝트 내 모든 세션을 볼 수 있음
- [ ] 일반 사용자 세션 뷰는 프로젝트 관리자 여부와 관계없이 본인의 세션만 표시
- [ ] 프로젝트 관리자가 프로젝트 내 다른 사용자의 세션을 종료할 수 있음
- [ ] 프로젝트 관리자가 모든 프로젝트 세션의 컨테이너 로그를 볼 수 있음
- [ ] 프로젝트 관리자가 모든 프로젝트 세션의 리소스 사용량을 볼 수 있음
- [ ] 프로젝트 관리자가 프로젝트 세션을 선택하여 일괄 작업을 수행할 수 있음
- [ ] 앱 런처, 터미널, SFTP, 컨테이너 커밋, 세션 이름 변경은 비소유 세션에 비활성화 상태 유지 (현재 superadmin 동작과 동일)
- [ ] 일반 사용자(프로젝트 멤버)는 여전히 자신의 세션만 볼 수 있음
- [ ] Superadmin 동작은 변경되지 않음

### 스토리 2: 프로젝트 관리자 — 모델 서비스 관리 (GF-31)

프로젝트 관리자는 할당된 프로젝트 내의 **모든 모델 서비스(배포)**를 조회하고 관리할 수 있습니다. 다른 사용자의 서비스에 대한 허용/불허 액션은 **현재 superadmin이 다른 사용자의 서비스에 대해 갖는 것과 동일**합니다 — 새로운 권한이 추가되거나 제거되지 않으며, 동일한 `created_user_email` 소유권 체크가 적용됩니다.

> **UI 배치**: 프로젝트 관리자의 모델 서비스 관리 기능은 Admin 메뉴의 전용 관리자 서비스 뷰에서 제공됩니다. 일반 사용자 페이지의 서비스 뷰는 프로젝트 관리자 여부와 관계없이 사용자 본인의 서비스만 표시합니다.

**기능 요구사항:**

1. **서비스 목록 가시성 (Admin 서비스 뷰)**
   - 프로젝트 관리자가 Admin 메뉴의 관리자 서비스 뷰에서 프로젝트 내 모든 모델 서비스 조회 (다른 사용자가 만든 것 포함)
   - 서비스 생성자를 구분할 수 있도록 소유자 정보 표시

   > **백엔드 확인 완료**: 세션과 동일하게 프로젝트 스코프로 서비스 조회 시 RBAC 기반 자동 필터링 적용. API 호출 전 권한 확인만을 위한 별도 요청은 불필요하며, 권한 검증은 백엔드 RBAC validator가 처리. (회의 2026-03-26 확인)

2. **서비스 생성**
   - 프로젝트 구성원이면 기본적으로 새 모델 서비스 배포 가능 (프로젝트 관리자 전용 기능이 아님)
   - **Admin 모드에서는 서비스 생성 불가**; 서비스 생성은 일반(사용자) 메뉴에서만 가능

3. **서비스 제어 작업** (참고: `EndpointList.tsx`, `EndpointDetailPage.tsx`)

   **다른 사용자의 서비스에 허용되는 작업** (현재 코드에서 소유권 체크 없음):
   - 목록에서 모델 서비스 **삭제/종료** 가능
   - 모든 모델 서비스의 **상세 정보 조회** 가능 (엔드포인트 상세 페이지)
   - 모든 모델 서비스에 대해 **토큰 생성** 가능
   - 모든 모델 서비스에 대해 **오토스케일링 룰 추가/삭제** 가능
   - 모든 모델 서비스에 대해 **라우트 동기화** 가능
   - 모든 모델 서비스에 대해 **에러 초기화** 가능
   - 서비스 정보 **새로고침** 가능
   - **서비스 설정 편집** (설정 버튼) — 다른 사용자의 서비스 설정 (레플리카 수, 이미지, 리소스 등) 수정 가능
   - **오토스케일링 룰 편집** (기존 룰의 설정 버튼) — 다른 사용자의 오토스케일링 룰 설정 수정 가능

   **다른 사용자의 서비스에 허용되지 않는 작업**:
   - (현재 superadmin과 동일한 수준으로 제한되는 추가 작업 없음)

4. **UI 동작**
   - Admin 메뉴의 모델 서비스 목록이 프로젝트 관리자 역할에 따라 표시 및 액션 활성화
   - 서비스 상세 페이지에서 허용된 범위 내에서 프로젝트 관리자에게 제어 기능 표시

**인수 기준:**
- [ ] 프로젝트 관리자가 Admin 서비스 뷰에서 프로젝트 내 모든 모델 서비스를 볼 수 있음
- [ ] 일반 사용자 서비스 뷰는 프로젝트 관리자 여부와 관계없이 본인의 서비스만 표시
- [ ] 프로젝트 관리자가 다른 사용자의 모델 서비스를 삭제/종료할 수 있음
- [ ] 프로젝트 관리자가 다른 사용자의 서비스에 대해 토큰 생성 및 오토스케일링 룰 관리가 가능함
- [ ] 프로젝트 관리자가 다른 사용자의 서비스 설정을 편집할 수 있음
- [ ] 서비스 생성은 Admin 메뉴에서는 불가하며 일반(사용자) 메뉴에서만 가능
- [ ] 일반 사용자는 여전히 자신의 서비스만 조회하고 제어할 수 있음
- [ ] Superadmin 동작은 변경되지 않음

### 스토리 3: 프로젝트 관리자 — 프로젝트 폴더 관리 (GF-32)

프로젝트 관리자는 할당된 프로젝트 내에서 **프로젝트 타입 폴더(가상 폴더)**에 대한 CRUD 작업을 수행할 수 있습니다. 여기서 "프로젝트 폴더"란 폴더 타입이 **프로젝트(group)**인 폴더를 의미하며, **사용자(user)** 타입 폴더와 구분됩니다. 세션, 서비스와 달리 이것은 **superadmin 동작과 동일하지 않습니다** — 프로젝트 관리자는 프로젝트 타입 폴더에 대해 새로운 관리 권한을 얻지만, 다른 사용자의 개인(사용자 타입) 폴더에 대한 접근 권한은 생기지 않습니다.

> **UI 배치**: 프로젝트 관리자의 폴더 관리 기능은 Admin 메뉴의 전용 관리자 폴더 뷰에서 제공됩니다. 일반 사용자 페이지의 폴더 뷰는 프로젝트 관리자 여부와 관계없이 기존과 동일하게 동작합니다.

**기능 요구사항:**

1. **폴더 가시성 (Admin 폴더 뷰)**
   - 프로젝트 관리자가 Admin 메뉴의 관리자 폴더 뷰에서 프로젝트 내 모든 **프로젝트 범위** 폴더 조회
   - 다른 사용자의 개인(사용자 범위) 폴더는 **보이지 않음** — 다른 사용자의 폴더에 대한 새로운 접근 권한 없음

2. **폴더 관리 작업 (프로젝트 폴더에 한함)**
   - 프로젝트 관리자가 새 프로젝트 폴더 **생성** 가능 — 일반(사용자) 메뉴와 Admin 폴더 뷰 **양쪽 모두에서 생성 가능**. 단, Admin 폴더 뷰에서는 **현재 선택된 프로젝트의 프로젝트 타입 폴더만** 생성 가능하도록 제한.
   - 프로젝트 관리자가 프로젝트 내 모든 프로젝트 폴더 **조회/목록** 가능
   - 프로젝트 관리자가 프로젝트 폴더 속성 **수정** 가능 (이름, 권한, 공유)
   - 프로젝트 관리자가 프로젝트 폴더 **삭제** 가능 (다른 사용자가 만든 것 포함)
   - 프로젝트 관리자가 **폴더 권한 관리** 가능 — 사용자 초대/제거, 접근 수준 변경

3. **프로젝트 폴더 크로스-프로젝트 공유**
   - **초기 버전에서는 제외. 추후 별도 이슈로 진행.**

4. **프로젝트 관리자가 할 수 없는 것**
   - **다른 사용자의 개인 폴더에 접근 불가** — 사용자 범위 폴더에 대한 가시성이나 제어 권한 없음
   - **폴더 소유권 변경 불가** — 프로젝트 폴더는 프로젝트 범위 유지

5. **UI 동작**
   - Admin 폴더 뷰에서 프로젝트 폴더에 생성·조회·수정·삭제 및 권한 관리용 액션 버튼 활성화
   - Admin 폴더 뷰의 폴더 생성은 현재 선택된 프로젝트의 프로젝트 타입 폴더로 제한
   - 일반(사용자) 폴더 뷰에서도 기존과 동일하게 프로젝트 폴더 생성 가능

**인수 기준:**
- [ ] 프로젝트 관리자가 일반(사용자) 폴더 뷰에서 프로젝트 폴더를 생성할 수 있음
- [ ] 프로젝트 관리자가 Admin 폴더 뷰에서 현재 프로젝트의 프로젝트 타입 폴더를 생성할 수 있음
- [ ] Admin 폴더 뷰에서의 폴더 생성이 현재 선택된 프로젝트의 프로젝트 타입으로 제한됨
- [ ] 프로젝트 관리자가 Admin 폴더 뷰에서 프로젝트 내 모든 프로젝트 폴더를 조회할 수 있음
- [ ] 프로젝트 관리자가 자신의 것뿐만 아니라 모든 프로젝트 폴더를 수정할 수 있음
- [ ] 프로젝트 관리자가 자신의 것뿐만 아니라 모든 프로젝트 폴더를 삭제할 수 있음
- [ ] 프로젝트 관리자가 모든 프로젝트 폴더의 권한을 관리 (사용자 초대/제거)할 수 있음
- [ ] 다른 사용자의 개인 폴더는 프로젝트 관리자에게 완전히 보이지 않음
- [ ] 프로젝트 관리자(프로젝트 어드민)뿐 아니라 도메인 관리자(도메인 어드민)도 다른 사용자의 개인(User 타입) 폴더를 조회할 수 없음
- [ ] 일반 사용자 폴더 뷰는 프로젝트 관리자 여부와 관계없이 기존과 동일하게 동작
- [ ] Superadmin 동작은 변경되지 않음

### 스토리 4: 프로젝트 관리자 — 프로젝트 멤버 조회

> **[확정] 초기 버전에서는 멤버 목록 조회만 제공. 멤버 추가/제거/역할 변경은 모두 초기 버전 범위 외.** (회의 2026-03-26 확정)
>
> 사용자 생성/추가/삭제를 프로젝트 관리자에게 허용할 경우 리소스 정책, 키페어, 크로스-프로젝트 접근 권한 등 복잡한 의존관계가 발생하여 초기 버전에서 제외하기로 결정. 워크로드(세션/서비스) 관리가 프로젝트 관리자의 핵심 요구이므로 우선 집중.

프로젝트 관리자는 할당된 프로젝트 내 **멤버 목록을 조회**할 수 있습니다. 사용자 생성, 프로젝트 추가/제거, 역할 변경 등 mutation 작업은 초기 버전에서 제외되며, superadmin 또는 도메인 관리자를 통해 처리합니다.

> **UI 배치**: 프로젝트 멤버 조회 기능은 Admin 메뉴 내에 "프로젝트 멤버" 섹션으로 제공됩니다.

**기능 요구사항 (초기 버전):**

1. **프로젝트 멤버 조회**
   - 프로젝트 관리자가 프로젝트에 소속된 사용자 목록 조회 가능
   - 멤버 목록에 사용자 이름, 이메일, 프로젝트 내 역할 표시

2. **프로젝트 관리자가 할 수 없는 것 (초기 버전)**
   - **사용자 생성 불가** — 리소스 정책·키페어 등 복잡한 의존관계로 인해 제외
   - **기존 사용자 프로젝트 추가 불가** — 크로스-프로젝트 접근 권한·프라이버시 이슈로 인해 제외
   - **멤버 제거 불가** — 초기 버전 범위 외
   - **멤버 역할 변경 불가** — 초기 버전 범위 외
   - **전체 사용자 조회 불가** — 전역 사용자 목록에 접근 불가; 프로젝트 멤버만 조회 가능
   - **사용자 프로필 수정 불가** — 이메일, 비밀번호 등 사용자 속성 변경 권한 없음

3. **UI 동작**
   - Admin 메뉴 내 "프로젝트 멤버" 섹션에 멤버 목록 조회 기능 제공
   - 멤버 추가/제거/역할 변경 버튼은 초기 버전에서 표시하지 않음

**인수 기준 (초기 버전):**
- [ ] 프로젝트 관리자가 프로젝트 내 모든 멤버를 조회할 수 있음
- [ ] 멤버 목록에 사용자 이름, 이메일, 프로젝트 내 역할이 표시됨
- [ ] 프로젝트 관리자가 전역 사용자 목록을 조회하거나 검색할 수 없음
- [ ] 사용자 생성/추가/제거/역할 변경 UI가 프로젝트 관리자에게 노출되지 않음
- [ ] Superadmin 동작은 변경되지 않음

## 공통 관심사

### 권한 확인 패턴

네 가지 스토리 모두 권한 기반 UI에 대한 공통 패턴을 공유합니다:

1. **페이지 로드 시**: `useCurrentUserProjectRoles()` 훅을 통해 사용자의 프로젝트 관리자 역할 확인 (`my_roles` 쿼리 사용)
2. **조건부 렌더링**: 감지된 역할에 따라 Admin 메뉴 내 관리자 수준의 제어 기능 표시/숨김
3. **API 호출**: 백엔드가 프로젝트 범위 결과를 반환하도록 적절한 파라미터 전달 (백엔드가 사용자의 RBAC 권한 기반으로 필터링)

### RBAC 백엔드 동작 방식 (회의 2026-03-26 확인)

- **API 호출 전 별도 권한 확인 요청(추가 round-trip) 불필요**: 백엔드가 모든 실제 요청(쿼리/mutation)에 대해 RBAC validator를 거쳐 권한을 검증. 권한이 없으면 에러 메시지 반환, 있으면 바로 동작하여 결과 반환. 단, **UI 노출 제어를 위한 역할 감지/조건부 렌더링은 유지** — 페이지 로드 시 사용자 역할을 감지하여 Admin 메뉴 및 관리자 제어의 표시/숨김을 결정하되, 실제 권한 판정은 항상 백엔드가 담당.
- **동일한 쿼리/mutation 사용**: superadmin이든 프로젝트 관리자든 동일한 쿼리/mutation을 호출. 백엔드가 요청자의 권한에 따라 결과를 필터링하여 반환.
- **레거시 API 확장 금지**: RBAC 기능을 위해 레거시 API를 확장하지 않음. RBAC 기반 신규 mutation이 준비되면 그때 기능을 지원.
- **도메인 관리자 권한 상속(cascade)**: 도메인 스코프의 권한은 하위 프로젝트·사용자에 대해서도 적용됨. 도메인 관리자는 도메인 내 모든 프로젝트 리소스를 조회/관리 가능.

### 개인 폴더 보안 (회의 2026-03-26 확인)

- **다른 사용자의 개인(User 타입) 폴더 내용(파일) 접근은 WebUI에서 어떤 역할에게도 허용되지 않음** — superadmin 포함. 폴더 목록(메타데이터)은 조회 가능하나, 폴더 내부 파일/데이터 열람은 불가.
- 이는 기업 보안 감사(audit) 요구사항에 의한 것: 비인가 데이터 접근 방지.
- 향후 허용하더라도 반드시 옵션(설정)으로 제공해야 함.

### 하위 호환성

- 백엔드가 아직 RBAC 역할 쿼리를 지원하지 않는 경우, WebUI는 기존 동작으로 폴백 (사용자가 자신의 리소스만 볼 수 있음)
- 기능 플래그 또는 백엔드 버전 감지(`@since` 디렉티브)로 새 기능을 게이트
- 기존 사용자 워크플로우가 깨지지 않아야 함

### RBAC 미지원 코어 버전 호환성

- `my_roles` 쿼리가 실패하거나 존재하지 않는 이전 코어 버전에서는 프로젝트 관리자 역할이 없는 것으로 처리하여 일반 사용자와 동일하게 동작
- 별도의 메뉴 표시 전략이 필요 없음 — 역할이 감지되지 않으면 Admin 관련 기능이 자연스럽게 비활성화됨

### 에러 처리

- 프로젝트 관리자가 권한이 없는 작업을 시도하는 경우 (예: 백엔드가 거부) 적절한 에러 메시지 표시
- 토큰/역할 만료를 우아하게 처리
